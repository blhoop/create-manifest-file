# Output Format — YAML Manifest

Defines the structure of the `.yml` file generated by this application.
The machine-readable version of this document lives in `server/config/outputSchema.js`.
**When adding, removing, or renaming fields — update both files.**

---

## File Naming

```
spokes/{spoke-name}.yml
```

The filename is for organization only. `subscription_name` is the source of truth for spoke identity.

---

## Structure Overview

```
[Subscription Block]   — one set of fields per file, defines the spoke
[Resources Block]      — array of resources to deploy or inventory
```

---

## Subscription Block

Appears once at the top of the file.

| Field | Required | Valid Values | Description |
|-------|----------|--------------|-------------|
| `subscription_name` | ✅ | Any string | Friendly name for the Azure subscription. Normalized to kebab-case for repo/workspace names and initials for resource names. |
| `environment` | ✅ | `dev` `test` `uat` `preprod` `prod` `lab` | Target deployment environment. |
| `default_location` | ✅ | See locations below | Default Azure region for all resources. Individual resources can override with their own `location` field. |
| `product_code` | | Short string (e.g. `lb`) | Short code used in all Azure resource names. Auto-derived from `subscription_name` initials if omitted. |
| `vnet_cidr` | | CIDR (e.g. `10.3.0.0/22`) | VNet CIDR block for this spoke's virtual network. Auto-allocated if omitted. |
| `subscription_id` | | UUID | Existing Azure subscription ID. If set, pipeline skips landing zone creation. Must pair with `spn_client_id`. |
| `spn_client_id` | | UUID | Existing SPN client ID for OIDC auth. Required when `subscription_id` is set. |

---

## Resources Block

A YAML array under the `resources:` key. Each entry defines one resource to deploy or inventory.

| Field | Required | Description |
|-------|----------|-------------|
| `name` | ✅ | Subsystem/component name (e.g. `web`, `booking-db`). Used in CAF resource naming. |
| `type` | ✅ | Resource type. See type list below. The first resource must be a compute type. |
| `location` | | Azure region override for this resource. Omit to use `default_location`. |
| `repo` | | Application source repo in `org/repo-name` format. When set, pipeline auto-generates CI/CD workflows. |
| `server_name` | | Parent SQL Server name — for `SQLDatabase` resources to link to their server. |
| `plan_name` | | Parent App Service Plan name — for `appserviceslots` or `webappslots` resources. |
| `function_app_name` | | Parent Function App name — for `functionappslots` resources. |
| `comments` | | Free-text hints that influence the manifest (e.g. `needs pgbouncer`, `serverless`, `zone redundant ha`). |

## Resource Types

### Builder Types
Generate new infrastructure. CAF-compliant names are auto-generated.

| Type | Description |
|------|-------------|
| `app_service` | Azure App Service (Web App) |
| `pg` | PostgreSQL Flexible Server |
| `cosmos` | Cosmos DB |
| `sql` | SQL Server + Database |
| `mysql` | MySQL Flexible Server |
| `sqlmi` | SQL Managed Instance |
| `aks` | Azure Kubernetes Service |
| `container_app` | Azure Container App |
| `vm` | Virtual Machine |
| `redis` | Azure Cache for Redis |
| `swa` | Static Web App (use `eastasia` — not available in `australiaeast`) |
| `key_vault` | Key Vault |
| `app_insights` | Application Insights |
| `container_registry` | Azure Container Registry |
| `service_bus` | Service Bus Namespace |
| `ai_foundry` | Azure AI Foundry |
| `ai_search` | Azure AI Search |

### Inventory Types
Document existing resources without generating new infrastructure.

| Type | Description |
|------|-------------|
| `WebApp` | Existing Web App |
| `FunctionApp` | Existing Function App |
| `AppServicePlan` | Existing App Service Plan |
| `StorageAccount` | Existing Storage Account |
| `SQLServer` | Existing SQL Server |
| `SQLDatabase` | Existing SQL Database |
| `KeyVault` | Existing Key Vault |
| `StaticSite` | Existing Static Web App |

| Rule | Location | Types |
|------|----------|-------|

## Complete Example

```yaml
# ---------------------------------------------------------------------------
# REQUIRED — Subscription identity
# ---------------------------------------------------------------------------
subscription_name: Order Book
environment: dev
default_location: australiaeast

# ---------------------------------------------------------------------------
# OPTIONAL — Overrides and existing infrastructure
# ---------------------------------------------------------------------------
product_code: ob
vnet_cidr: 10.3.0.0/22
subscription_id: 04a33693-83cd-41d8-b1bc-a67a3476a439
spn_client_id: 523b2cb8-04f0-448f-8630-0736b1a2e5bb

# ---------------------------------------------------------------------------
# RESOURCES
# ---------------------------------------------------------------------------
resources:
  - name: web
    type: app_service
    location: australiaeast
    repo: MyOrg/ob-app
    comments: main web app serving the Order Book UI

  - name: notifications
    type: app_service
    repo: MyOrg/ob-func-app
    comments: function apps for notifications

  - name: booking-db
    type: pg
    comments: needs pgbouncer

  - name: reporting-server
    type: SQLServer

  - name: reporting-db
    type: SQLDatabase
    server_name: reporting-server

  - name: cache
    type: redis

  - name: portal
    type: swa
    location: eastasia
    comments: SWA not available in australiaeast

  - name: config-site
    type: swa
    location: eastasia
    repo: MyOrg/ob-config-app
```

## Notes

- `subscription_name` is normalized by the downstream builder:
  - kebab-case → used for repo and workspace names (e.g. `lightning-book`)
  - initials → used in all Azure resource names (e.g. `lb`)
- `product_code` overrides the auto-derived initials when set explicitly
- The first entry in `resources` must be a compute type: `app_service`, `aks`, `container_app`, or `vm`
- Optional fields are omitted entirely from the output when empty (not written as empty strings)
- **Parent resource references** (`server_name`, `plan_name`, `function_app_name`):
  - Use `server_name` to link a `SQLDatabase` to its parent `SQLServer`
  - Use `plan_name` to link `appserviceslots` or `webappslots` to their parent `AppServicePlan`
  - Use `function_app_name` to link `functionappslots` to their parent `FunctionApp`
  - Terraform consumes these to build the proper resource dependencies
